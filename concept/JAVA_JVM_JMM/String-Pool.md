# String Pool

## What I know so far

Strings are immutable objects.

Creating identical immutable objects repeatedly wastes memory.

To solve this problem, the JVM maintains a **String Pool**, a special mechanism inside the Heap that stores one canonical instance of every String literal.

Instead of creating duplicate String objects, identical literals reuse the same object.

This reduces memory usage while preserving Java's object model.

---

# Why does this problem exist?

Imagine writing

```java
String s1 = "Hello";
String s2 = "Hello";
String s3 = "Hello";
```

Without any optimization, the Heap would contain

```
Heap

"Hello"

"Hello"

"Hello"
```

Three identical immutable objects.

Nothing can ever change their contents because Strings are immutable.

Creating three separate objects wastes memory.

JVM engineers asked a simple question.

> Why create multiple immutable objects with identical values?

Instead,

store one object.

Allow everyone to share it.

```
           "Hello"

              ▲
              │
      ┌───────┼───────┐
      │       │       │
     s1      s2      s3
```

One object.

Three references.

---

# Engineering Mental Model

Imagine a public library.

Without a library

```
Person A buys Book

Person B buys Book

Person C buys Book
```

Three identical books.

Large memory waste.

With a library

```
Shared Book

      ▲

      │

A     B     C
```

Everyone reads the same immutable book.

Nobody modifies it.

Exactly how String Pool works.

# Where does the String Pool live?

Many developers think

```
Runtime Data Areas

Heap

Stack

Metaspace

String Pool
```

This is incorrect for modern JVMs.

The String Pool is **not** another Runtime Memory Area.

It is a managed region inside the Heap.

```
Heap

├── Student Objects

├── Order Objects

├── Employee Objects

└── String Pool
      ├── "Hello"
      ├── "Java"
      └── "Spring"
```

# String Literal

```java
String s = "Hello";
```

Execution

```
Compiler

↓

String Literal

↓

JVM checks String Pool

↓

Already Exists?

      │

Yes ─────────────► Reuse Object

No ──────────────► Create Object

↓

Reference assigned to s
```

Every literal first checks the String Pool.

---

# new String()

```java
String s = new String("Hello");
```

Many people misunderstand this.

Execution is

```
Resolve Literal

↓

String Pool

↓

new

↓

Create New Heap Object

↓

Assign Reference
```

Notice

The String Pool object still exists.

The `new` keyword explicitly requests a **new object**, even if an identical String already exists.

This preserves one of Java's most important guarantees.

> Every use of `new` creates a new object.

---

# intern()

Suppose

```java
String s = new StringBuilder()
        .append("Hel")
        .append("lo")
        .toString();
```

This String was not created from a literal.

It simply exists as a Heap object.

Calling

```java
s.intern();
```

asks the JVM

```
Does String Pool already contain

"Hello" ?

        │

Yes

↓

Return pooled reference

No

↓

Add canonical String

↓

Return pooled reference
```

Notice

`intern()` does **not** convert the current object into a pooled object.

It returns the **canonical String** for that value.

---

# == vs equals()

Suppose

```java
String s1 = "Java";

String s2 = "Java";

String s3 = new String("Java");
```

Memory

```
Stack

s1 ─────┐
         │
s2 ──────┘
         │
         ▼

String Pool

"Java"

s3

↓

Heap

"Java"
```

Comparison

```
s1 == s2

↓

true
```

Same object.

```
s1 == s3

↓

false
```

Different objects.

```
s1.equals(s3)

↓

true
```

Same contents.

# Compile-Time Optimization

Suppose

```java
String s = "Hel" + "lo";
```

The compiler already knows

```
"Hel"

+

"lo"

↓

"Hello"
```

Before the program even runs.

The generated bytecode behaves as if the source code had been

```java
String s = "Hello";
```

No runtime concatenation occurs.

---

Suppose instead

```java
String part = "Hel";

String s = part + "lo";
```

Now

`part`

is a variable.

The compiler cannot determine the final value during compilation.

Runtime concatenation creates a new String object.

---

# Why is String immutable?

Sharing would be impossible if Strings could change.

Imagine

```
s1

↓

"Hello"

↑

s2
```

Now

```java
s1.setValue("Bye");
```

Suddenly

```
s2

↓

"Bye"
```

Every reference would unexpectedly observe the modification.

Immutability makes safe sharing possible.

# Common Interview Misconceptions

### ❌ String Pool is another Runtime Memory Area.

✅ Modern JVM stores interned Strings inside the Heap.

---

### ❌ `new String()` uses the String Pool.

✅ `new` always creates a new Heap object.

---

### ❌ `intern()` creates a new object.

✅ It returns the canonical pooled reference.

---

### ❌ `==` compares String contents.

✅ `==` compares object references.

---

### ❌ `equals()` compares addresses.

✅ `equals()` compares character contents.

---

### ❌ Every String is pooled.

✅ Only interned Strings and String literals are pooled.

---

# Decision Checklist

Whenever working with Strings, ask

```
□ Is this a String literal?

□ Is this created using new?

□ Do I actually need a new object?

□ Am I comparing references or contents?

□ Could String Pool reuse this object?

□ Is intern() useful here?
```

# Engineering Patterns

The String Pool follows the same JVM philosophy.

```
Observe

↓

Strings are immutable

↓

Identical literals appear frequently

↓

Store one canonical object

↓

Reuse it everywhere
```

Notice the recurring JVM engineering mindset.

```
Garbage Collection

↓

Most objects die young.

----------------------------

Class Loader

↓

Load only required classes.

----------------------------

JIT

↓

Compile only hot code.

----------------------------

String Pool

↓

Share immutable objects.
```

Instead of optimizing everything,

the JVM identifies common patterns in real applications and optimizes those patterns while preserving correctness.
